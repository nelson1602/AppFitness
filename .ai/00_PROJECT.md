# AppFitness Project Constitution

Version: 1.0
Status: Active
Last Updated: 2026-07-03

---

# Purpose

AppFitness is a production-grade cross-platform mobile application designed to help users improve their health, nutrition, physical condition, and long-term wellness through deterministic coaching, medical data tracking, and evidence-based recommendations.

The application must provide a fast, secure, offline-first experience while maintaining enterprise-level software quality, scalability, maintainability, and regulatory compliance.

The project is intended for long-term evolution and must be designed to support future capabilities without requiring major architectural rewrites.

---

# Vision

Build one of the most reliable and technically robust personal fitness platforms by combining:

- Medical evaluation tracking
- Physical progress monitoring
- Deterministic coaching
- Intelligent recommendations
- Offline-first architecture
- Modern mobile user experience
- Enterprise software engineering practices

The platform should assist users in making informed decisions based on measurable data rather than assumptions.

---

# Core Principles

Every technical decision must support the following principles.

## Reliability

The application must behave predictably.

Identical inputs must always generate identical outputs.

No hidden side effects are allowed.

---

## Determinism

All coaching recommendations generated by the local iCoach Engine must be deterministic.

Business rules must never rely on randomness.

AI-generated recommendations may exist in the future but must always remain optional and clearly separated from deterministic calculations.

---

## Offline First

The mobile application must remain fully functional without Internet connectivity for up to 48 hours.

The local SQLite database is considered the primary operational data source.

Synchronization with the backend occurs asynchronously whenever connectivity becomes available.

---

## User Ownership

Users own their health data.

The application exists to assist—not replace—medical professionals.

Medical decisions always belong to licensed healthcare providers.

The application must never present itself as a medical diagnosis tool.

---

## Security by Design

Security is not an additional feature.

It is a mandatory architectural requirement.

Sensitive information must always be protected both in transit and at rest.

---

## Privacy by Design

Personal health information must be handled according to modern privacy standards.

The architecture should remain compatible with:

- HIPAA principles
- GDPR principles
- Privacy-by-Design
- Least Privilege

---

## Scalability

Every new feature must be designed to scale.

Avoid architectural shortcuts that create technical debt.

---

## Maintainability

Code should be understandable by engineers unfamiliar with the project.

Readability is preferred over cleverness.

---

## Simplicity

Choose the simplest solution that satisfies the requirements.

Avoid unnecessary abstractions.

Avoid premature optimization.

# iCoach Philosophy

The iCoach Engine is the heart of AppFitness.

It consists of two independent systems.

## Deterministic Engine

Runs entirely on-device.

Responsible for:

- Nutrition calculations
- Workout generation
- Progress evaluation
- Medical restrictions
- Calorie calculations
- Macro calculations
- Exercise progression

Every output must be mathematically reproducible.

---

## Knowledge Engine

Runs only when Internet connectivity exists.

Its purpose is to gather updated scientific information.

It never makes decisions directly.

Instead, it produces validated knowledge that can later improve deterministic rules after review.

---

# AI Strategy

Artificial Intelligence is considered an assistant—not an authority.

AI should:

- Explain
- Suggest
- Summarize
- Educate

AI must never replace deterministic business rules.

AI-generated recommendations must always be distinguishable from deterministic recommendations.

---

# Offline Strategy

Local SQLite database:

Primary source of truth while offline.

Backend PostgreSQL:

System of record after synchronization.

Synchronization must be:

Incremental

Conflict-aware

Retryable

Auditable

Non-blocking
